## Quest Narrative

The Guild Hall README currently reads like internal documentation — a feature status checklist, duplicated setup sections, and a full project tree dump. Meanwhile, the campaign-mode README demonstrates what a good project README looks like: it opens with a hook, speaks to the reader, and uses progressive disclosure to reveal technical depth only when the reader is ready for it. This quest is about rewriting the Guild Hall README to sell the adventure, not the spreadsheet — and ensuring every claim in it is accurate and current.

## Success Criteria

1. **Hook first** — Opens with a compelling value proposition that makes someone want to keep reading, not a version number
2. **Reader-centred voice** — Uses "you" language throughout; speaks to what the reader gets, not what the app has
3. **Progressive disclosure** — Flows from why → what → how → get started; technical detail comes late, not early
4. **Show, don't list** — Includes a concrete scenario showing what using Guild Hall *feels like*
5. **Clean structure** — No duplicate sections, no feature status checklists, no project tree dump; links out to detailed docs instead of inlining them
6. **Mock-ups preserved** — Preview images are kept but moved to a separate documentation page, linked from the README
7. **Accuracy verified** — All information in the README and any new pages is reviewed against the actual codebase to ensure it is relevant and up to date

## Anticipated Dragon

The temptation to keep everything in — to treat the README as a comprehensive reference rather than a front door. Cutting content feels like losing information, even when moving it somewhere better.
